# Wymagania projektowe

## 1. Przypadki użycia
- Minimum 8 przypadków użycia, w tym co najmniej 3 złożone
- Każdy przypadek użycia musi zawierać:
  - aktora
  - scenariusz główny
  - scenariusze alternatywne
  - warunki początkowe i końcowe
- Każdy przypadek użycia musi mieć przypisany priorytet

## 2. Weryfikowalność
Każdy przypadek użycia musi być powiązany z:
- implementacją (plik/moduł)
- co najmniej jednym testem

Tabela powiązań jest obowiązkowa:
| Przypadek użycia | Moduł | Test |

## 3. Modelowanie UML

### Diagram klas
- Minimum jeden diagram globalny
- Spójność z implementacją jest obowiązkowa
- Wymagane relacje UML

### Diagramy interakcji
- Minimum 3 diagramy
- Każdy powiązany z przypadkiem użycia
- Musi odzwierciedlać rzeczywiste metody w kodzie

## 4. Architektura
- Opis architektury systemu
- Uzasadnienie wyboru

## 5. Implementacja
- Implementacja wszystkich przypadków użycia
- Modularność kodu
- Dobre praktyki

## 6. Testy i metryki
- Testy pokrywające przypadki użycia
- Wymagane:
  - pokrycie kodu ≥ 60%
  - + co najmniej jedna dodatkowa metryka (np. złożoność)

## 7. Definicja zakończenia
Projekt uznaje się za ukończony, jeśli:
- wszystkie przypadki użycia są zaimplementowane
- testy przechodzą
- pokrycie kodu ≥ 60%
- dokumentacja kompletna

## 8. Kryteria oceny
- wymagania (20%)
- UML i architektura (20%)
- implementacja (30%)
- testy i metryki (20%)
- SCRUM i organizacja pracy (10%)
